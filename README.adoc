== Concepts and terminology
Some key terms for better understanding taken from the https://ziglang.org/documentation/0.16.0/#Compilation-Model[compilation model]:

* *Zig module:* Any valid Zig file, such as a namespace, or implicit struct, etc.
* *Manifest:* The `build.zig.zon` file in the project root.
* *Build system script:* The `build.zig` module.
** This module works similar to an executable, so it’s basically an entry point with the reserved `build()` public function.
* *Libraries:* Modules that are meant to be imported by other modules, these don’t have a `main()` public function, and by convention are `usually named `root.zig`, but this is not a strict requirement. When `scanned, the linter will crawl around all the public reachable `declarations from the entry point or root source file, *this behavior is `shared across all module types.*
* *Executables:* Executables are those modules that have the special public function `main()`, a module with that function is strictly considered an executable, treating it as its entry point.
* *Test modules:* Test modules are similar to libraries, any `test` declaration presence will mark the module as a test module, Zig allows to define special test modules too.
* *Documentation comments:*
** Standard: `///`
** Top-level: `//!`

== Scanning strategy
=== Default scanning without path arguments
Whenever the CLI is run without any path arguments, the current working directory is scanned. The scanning strategy will attempt to find both build system script and related files to resolve the project structure and dependencies.

==== Build system script scanning
It expects a manifest file to be present in your current working directory, then scans the configured `.paths` entries.

Path dependencies listed in `build.zig.zon` under `.dependencies` (each entry with a `.path`) are *excluded from linting by default*, similar to how Rust tooling skips crates.io dependencies while linting your workspace crate. A `.paths` entry may still list `modules` for packaging; Docent uses dependency metadata to avoid linting vendored trees under those paths.

* CLI: pass `--lint-dependencies` to include dependency directories (including when you pass an explicit path under a dependency root).
* Library: `docent.targeting.Options{ .lint_dependencies = true, .exclude_roots = ... }`.
* Build step: `docent.addLintStep(b, .{ .lint_dependencies = true, ... })`.

Dependency roots are resolved from the nearest `build.zig.zon` and passed as `exclude_roots` automatically when you run the CLI with no path arguments.

.Directory behavior:
* If a directory contains `root.zig`, Docent treats it as an entrypoint and lints files that are publicly reachable from that root via `pub const ... = @import("...")` chains.
* If a directory has no `root.zig`, Docent treats each top-level `.zig` file as a module entrypoint and lints the union of their publicly reachable files.
* If no top-level `.zig` files exist, Docent falls back to linting all `.zig` files in that directory tree.

.Reachability notes:
* Traversal is recursive across imported files, so multi-hop public chains are included.
* Imports reachable only through non-public declarations are excluded.
* Package imports (for example `@import("std")`) are not treated as local lint targets.
* `build.zig` and files under `build/` are ignored by default.
** Instead of files within the build directory, it’s mostly all of those files that are used/imported by the main build script (`build.zig`).
* This avoids false-positive API checks from build tooling paths that are commonly present in `build.zig.zon` `.paths`.
* CLI users can opt in with `--include-build-scripts`.
* Library users can opt in via `docent.targeting.Options{ .include_build_scripts = true }`.
* Build integration users can opt in via `docent.addLintStep(..., .{ .include_build_scripts = true, ... })`.

.Dependency exclusion:
* Files under `.dependencies.*.path` roots are skipped unless `lint_dependencies` is enabled (see above).
* Pass extra directory roots via `docent.targeting.Options.exclude_roots` when you need additional ignores beyond manifest dependencies.

==== Scanning with arguments
When explicit paths are passed, each argument is treated as a file or directory target.

.Supported cases include:
* A single file.
* A single directory.
* A mix of files and directories.

When explicit paths are provided, the default project discovery is disabled and only the specified targets are scanned.

.Path validity
* Explicit paths are validated strictly by default.
* If a path does not exist, cannot be accessed, or refers to a non-Zig module, an error is emitted and the process exits with a non-zero status.
* If a directory is valid but does not yield any Zig lint targets, a warning is emitted and that directory is skipped.
* If all explicit targets are valid but none yield any Zig lint targets, the process exits with a non-zero status.

.Path resolution
* If the target is a Zig source file, it is treated as a module entry point candidate.
* If the target is a directory, module entry points are resolved within that directory.
* For each resolved file or directory, the module kind is determined:
** A file containing a public `main()` is treated as an executable entry point.
** A file containing a public `build()` is treated as a build script entry point.
** Otherwise, it is treated as a library entry point.
* In all cases, the publicly reachable API is traversed starting from the resolved entry point.

== Severity levels
All rules accept one of these levels:

* *Allow:* Rule is disabled. No diagnostics are emitted.
* *Warn:* Diagnostics are emitted, but they do not cause the process to exit with an error code.
* *Deny:* Diagnostics are emitted, and the process exits with an error code.
* *Forbid:* Similar to "`Deny`", but the rule cannot be overriden by a subsequent configuration.

The distinction between _Deny_ and _Forbid_ matter for locking a rule in CI regardless of any local flag overrides. For example, setting _Forbid_ in the manifest cannot be weakened to any other level in the command line.

== Documentation rules
=== Missing documentation comment (`missing_doc_comment`)
Checks (_warned_ by default) for missing https://ziglang.org/documentation/0.16.0/#Doc-Comments[documentation comments] on public declarations. Including, but not limited to:

* https://ziglang.org/documentation/0.16.0/#Functions[Functions and its parameters]
* https://ziglang.org/documentation/0.16.0/#Variables[Global variables]
* https://ziglang.org/documentation/0.16.0/#enum[Enumerations and its fields]
* https://ziglang.org/documentation/0.16.0/#struct[Structures and its fields]
* https://ziglang.org/documentation/0.16.0/#union[Unions and its fields]
* https://ziglang.org/documentation/0.16.0/#Errors[Error sets]
** Error set fields aren’t checked since Zig documentation generator doesn’t support them.

==== Re-exported declarations
Public re-exports are resolved transitively.

For `pub const Foo = @import("other.zig").Bar`, the rule follows the re-export chain and checks the documentation on the final resolved declaration.

* If the resolved declaration is documented, no diagnostic is emitted.
* If it is undocumented, the diagnostic points to the resolved declaration.
* If resolution fails, the re-export is skipped to avoid false positives.

=== Empty documentation comment (`empty_doc_comment`)
Checks (_warned_ by default) for comments that are present but blank/empty.

For guidance on how to write good documentation comments, see https://ziglang.org/documentation/0.16.0/#Doc-Comment-Guidance[Zig’s documentation comment guidance].

=== Missing documentation test (`missing_doctest`)
Checks (_allowed_ by default) for https://ziglang.org/documentation/0.16.0/#Doctests[documentation test] coverage.

.What it checks:
* Collects top-level `pub fn name(...)` declarations.
* Collects identifier-style tests `test name { ... }`.
* Emits a diagnostic when a public function has no matching identifier-style doctest.

.Notes:
* String-literal test names (for example `test "name"`) are not counted
as doctests for this rule.

=== Private documentation test (`private_doctest`)
Checks that identifier-style doctests reference public symbols.

.What it checks:
* Collects top-level public function names and public variable/constant names.
* For each `test name { ... }`, emits a diagnostic if `name` is not public.

=== Documentation test naming mismatch (`doctest_naming_mismatch`)
Checks for style mismatch when a doctest name matches a public function but is written as a string literal.

.What it checks:
* If `pub fn foo(...)` exists and the file uses `test "foo"`, it suggests using `test foo`.

==== Future refactoring due to extension proposal
There’s an open proposal to extend the syntax of document tests by Mitchell Hashimoto: https://github.com/ziglang/zig/issues/19518. It aims to allow multiple document test declarations, each one having its own description.

=== Missing top-level documentation comment (`missing_top_level_doc_comment`)
Checks (_warned_ by default) for missing https://ziglang.org/documentation/0.16.0/#Top-Level-Doc-Comments[top-level documentation comments].

.What it checks:
* File-level module doc comment (`//!`) at the start of a library entry point file (the implicit module container). Named `pub const` structs, enums, and unions use `///` doc comments instead; inner `//!` is not required or expected on those types.

.This rule works as an extension of the following rules:
* _Missing documentation comment,_ because the comment is missing.
* _Empty documentation comment,_ if the comment is present but blank.

==== Future refactoring due to removal proposal
Top-level comments are marked as an https://codeberg.org/ziglang/zig/issues/30132[_urgent proposal_ for removal]. If it’s removed, this rule will be refactored to the respective alternative.

=== Missing entry point documentation comment (`missing_entry_point_doc_comment`)
Checks (_warned_ by default) for missing documentation comments on the entry point of a module.

This is specially relevant for libraries or public package modules, where a library has a single entry point (the root source file), and this rule ensures that the entry point itself is documented.

.The rule works as an extension of the following rules:
* _Missing documentation comment,_ because the comment is missing.
* _Empty documentation comment,_ if the comment is present but blank.
* _Missing top-level documentation comment,_ because the type of documentation comment for the entry point is a top-level one.

=== Starts with _module_
Checks that a module entry-point doc comment starts with _Module_. Inspired by https://github.com/godoc-lint/godoc-lint#pkg-doc[Godoc-Lint's `pkg-doc` rule].

For example, let's consider https://github.com/jassielof/carnaval[Carnaval's] module entry point (`src/lib/root.zig`).

.Compliant
[source,zig]
----
//! Module carnaval provides style definitions and utilities for formatting output in the terminal.
----

.Non-compliant
[source,zig]
----
//! This module provides style definitions and utilities for formatting output in the terminal.
----

== Style rules
=== Naming conventions
Naming conventions for declarations and module (file names) follow the https://ziglang.org/documentation/0.16.0/#Names[Zig documentation].

.With the following clarifications:
* Error set values are expected to be snake_cased, even though the standard library uses PascalCase for them, they technically fall under the value category, similar to enumeration values. And Zig itself isn't quite consistent with https://ziglang.org/documentation/0.16.0/std/#std.builtin.OptimizeMode[its own naming conventions]

==== Tiger Beetle's Tiger Style

Tiger Beetle has its own style convention called https://tigerstyle.dev/[Tiger Style], and it's a nice reference point to consider for _experience_:

* https://tigerstyle.dev/#nouns-and-verbs[Noun and Verbs]
* https://github.com/tigerbeetle/tigerbeetle/blob/main/docs/TIGER_STYLE.md#naming-things[Naming things (Markdown version, more detailed)]

It's also worth considering for its 3 main points: _Safety, Performance, and Experience_.
