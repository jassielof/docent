= Linting Rules

== Doc comments

=== Broken intra-doc links

Warns on broken intra-doc links.

NOTE: Given that Zig's doc comment support for Markdown is ambiguous and its specification is yet to be fully defined, this rule won't be implemented until it's clearer. While intra-doc links work by using backticks to reference items, this behavior isn't documented yet. See https://github.com/ziglang/zig/issues/19866.

== Style

=== Identifier case

Checks for consistent identifier naming case conventions, by default it follows the {zig-docs}#Names[Zig documentation style]. It is reported by `docent check style`. Like the complexity checks, it measures *every* identifier — both public and non-public. File discovery starts from the project's module roots on the public API surface, then follows a *reachability traversal* that includes every local `@import("…zig")` (including private imports such as `const Severity = @import("../../severity.zig");`); within each reached file no visibility filter is applied.

Checks are:
`snake_case`::
* Namespaces, considering that every Zig source file is basically a structure, to ease the differentiation between a structure file and a namespace file, without depending on the file name case or its identifier, if a source file doesn't have structure fields, it's automatically marked as a namespace.
* Global variables and constants
* Enumerators
* Fields or values from structures, union, and error sets.
`camelCase`::
* Functions (concrete ones).
`PascalCase`::
* Structures
* Unions
* Enumerations
* Error sets
* Generic functions.

[NOTE]
====
* Error set values technically fall under the value category, similar to enumerators. And Zig itself isn't quite consistent with {zig-docs}std/#std.builtin.OptimizeMode[its own conventions]. Regardless, we'll follow the standard library conventions, so error-set values are expected to be `PascalCase` (e.g. `error.OutOfMemory`).
* It must be careful on re-exported declarations that work as aliases, for example a function that is re-exported in the module root, but lives in a different place, it's a global constant, but internally, it's a function being aliased to that constant. Mostly to avoid false positives, this causes `camelCase` global constants that are in fact just aliases to functions. Declarations whose initializer is a plain identifier, field access, call, or `@import(...)` are therefore skipped.
* Generic functions are detected as those returning `type`.
* Quoted identifiers (`@"foo bar"`) are exempt.
* `@import("…zig")` paths: when the resolved file is a namespace (no structure fields at file scope), the basename must be `snake_case` (e.g. `reachability.zig`, suggested as `diagnostic_message.zig` from `DiagnosticMessage.zig`). The import binding must also be `snake_case` for whole-module imports (e.g. `const severity = @import("severity.zig");`, not `const Severity = …`). Member re-exports (`pub const SeverityLevel = @import("severity.zig").Level;`) name the imported declaration and are not checked as namespace bindings. Struct-at-file-scope modules (e.g. `Diagnostic.zig`, `LintResult.zig`) keep `PascalCase` basenames and bindings.
====

=== Function size

Function size is measured in terms of physical true LoCs, which excludes comments, blank lines, function declaration, and braces.

.Affected by the following factors:
* Comments
* Blank lines
* True LoC, the code per se
* Braces (Zig's defaults to inline opening brace and newline closing brace)
* Function signature, the whole declaration of the function (e.g. `fn hello()`)

Logical LoCs, which counts the number of statements or expressions, isn't considered here, as there are already complexity checks (<<_cyclomatic_complexity>> and <<_cognitive_complexity>>).

NOTE: This doesn't mean that it will never be implemented, but it's implementation effort will only be considered on demand and contribution.

== Complexity

=== Cyclomatic Complexity

Defaults to a limit of 10 linearly independent paths, as suggested by <<mccabe-guidance>>. A score of 10–15 is considered _complex_, and anything above 15 is considered _risky_.

NOTE: Unlike tools like Lizard—which can be configured to treat an entire `switch` statement as a single branch—this tool enforces the traditional mathematical definition of _Cyclomatic Complexity_. While some teams modify CC to act as a proxy for readability, it should be used strictly as a **testability metric** (mapping directly to the number of required unit tests). Readability and maintainability are instead handled by **Cognitive Complexity**, which provides a much more reliable metric for human code comprehension.

.Edge cases:
* Exhaustive `switch` statements over `enum`s are still penalized, as each branch represents a real, testable path regardless of whether the compiler requires it.

.Options:
* The threshold can be configured with whole numbers.

=== Cognitive Complexity

Defaults to the Sonar-recommended threshold of 15. See <<sonar-reasoning>> and <<sonar-paper>>. A function is flagged when its score is *strictly greater* than the threshold. It is reported by `docent check complexity`.

Unlike the doc-comment rules, complexity checks measure *every* function — both public and non-public. File discovery still starts from the project's module roots and follows the public API surface, but within each reached file no visibility filter is applied.

Unlike <<_cyclomatic_complexity>>, the score is not a count of paths but an estimate of how hard the control flow is to *understand*. It accumulates three kinds of increments per function:

Structural::
Control-flow structures that also receive a nesting increment: `if`, `for`, `while`, `switch`, and `catch`. Each adds `1 + current_nesting`, so the same structure costs more the deeper it is nested.
Hybrid::
`else` and `else if` add `1` but never receive a nesting increment (the cost of reading the `if` has already been paid), yet they still raise the nesting level for their body.
Fundamental::
Breaks in linear flow that are independent of nesting: each new sequence of binary logical operators (`and` / `or`), a labeled loop `break`/`continue`, and direct recursion.

.Zig-specific mappings:
* `catch` is treated like an exception `catch` clause (structural). `orelse` is the optional counterpart of null-coalescing and is intentionally ignored.
* A labeled `break`/`continue` only increments when the label targets a `for`/`while` loop. Labeled block breaks (`break :blk value`) are value expressions and are not penalized.
* Only direct recursion (a function calling itself by name) is scored; indirect recursion is out of scope.
* `switch` and all of its prongs combined incur a single structural increment, matching the Sonar treatment of `switch`.

.Options
* The threshold can be configured with whole numbers in project config under `complexity.cognitive_threshold`; it defaults to 15.
* The severity (`cognitive_complexity`) follows the usual rule severities and can be set in project config.

=== Maximum function parameter

The maximum number of parameters in many tools and teams often default to around 5, to encourage simpler APIs. In Zig, however, it is common to pass interface parameters such as allocators, writers, and I/O interfaces as explicit dependencies. To be gentler on Zig codebases and more focused on true domain-specific parameters, the default limit here is set to 7.

A function is flagged when its parameter count is *strictly greater* than the threshold. It is reported by `docent check complexity`. Like the other complexity checks, it measures *every* function in the import-closure reachable from the module roots.

.Options
* The threshold can be configured with whole numbers in project config under `complexity.max_fun_params_threshold`; it defaults to 7.
* The severity (`max_fun_params`) follows the usual rule severities and can be set in project config.

== Configuration of rule options

Configuration options for the rules are defined in the project root, under the `.config` directory, in the file `docent.toml`. Run `docent init` to write a starter file with the published JSON Schema URL for editor validation. Docent does not use Zig's ZON notation for project config.

When no config file is found, or a rule is omitted, Docent uses the `RuleSet` defaults from the library.

.`docent.toml`:
====
[source,toml]
----
[docs.missing_doc_comment]
level = "deny"
check_parameters = false

[docs]
missing_doctest = "allow"
private_doctest = "warn"

[complexity.cognitive_complexity]
level = "warn"
threshold = 15

[complexity.max_function_parameters]
level = "allow"
threshold = 7
----
====

Rule severities can be set as a string shorthand in a section table (for example `missing_doctest = "allow"` under `[docs]`) or as a nested table with `level = "warn"`. Severity values are `allow`, `warn`, `deny`, and `forbid`. A `forbid` level cannot be relaxed by a later entry in the same file.

Nested rule tables hold per-rule options. Under `[docs.missing_doc_comment]`, set `check_parameters = true` to make `missing_doc_comment` also require `///` on each named function parameter (disabled by default).

Under `[complexity]`, rules such as `cognitive_complexity` and `max_function_parameters` accept a `threshold` alongside `level`. Omitted options fall back to the library defaults.

.Options:
* You can specify a custom path for the configuration file with `--config-path PATH` on `docent check`, `docent status`, and related commands.
