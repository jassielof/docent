= Linting Rules

Several rules are inspired by established Go documentation conventions, particularly those enforced by <<godoc-comments>> and <<godoc-lint>>. Go's standard library is notable for its strict, consistent documentation guidelines, and its tooling ecosystem — especially <<golangci-lint>>'s aggregation approach — reflects a culture of broad, opinionated quality checks across style, complexity, and documentation simultaneously.

Rust's Rustdoc and Clippy also informed specific rules here, though their documentation linting tends to focus narrower: missing docs and broken intra-links, etc.. Both ecosystems influenced Docent's rule design.

While default rules focus on documentation comments, style and complexity rules are also available as explicit sub-commands. The reasoning is mostly separation of concerns, as well as keeping consistency with other ecosystem's tooling.

[NOTE]
.Severity and examples
====
* For severity levels, see the link:{website}lib/#docent.SeverityLevel[library documentation].
* Each rule description starts with the verb of the severity level, for example, _"Warns on...", "Allows..."_, etc...
* For examples, see <<_test_suite>>.

== Doc comments

By default, these rules will be enforced across the public API surface. One can also configure the rules selectively or completely if desired to consider private declarations as well.

=== Missing doc comment

Warns on missing doc comments on public declarations. Including, but not limited to:

* {zig-docs}#Functions[Functions and its parameters]
** Parameters are not checked by default. Enable `docs.require_function_param_docs` in project config to require `///` on each named parameter of checked functions.
* {zig-docs}#Variables[Global variables]
* {zig-docs}#enum[Enumerations and its fields]
* {zig-docs}#struct[Structures and its fields]
* {zig-docs}#union[Unions and its fields]
* {zig-docs}#Errors[Error sets]
** Error set fields aren’t checked since Zig documentation generator doesn’t support them.

.Options
* Function parameters are optional to document by default. Set `docs.require_function_param_docs` to `true` in project config to require `///` on each named parameter of functions that are already checked by this rule.

==== Re-exported declarations

Public re-exports are resolved transitively.

For `pub const Foo = @import("other.zig").Bar`, the rule follows the re-export chain and checks the documentation on the final resolved declaration.

* If the resolved declaration is documented, no diagnostic is emitted.
* If it is undocumented, the diagnostic points to the resolved declaration.
* If resolution fails, the re-export is skipped to avoid false positives.

==== Specializations

To help with clarifications and context, the following cases are reported with a more specific diagnostic message.

Missing module doc comment::
Warns on missing doc comments on the module root source file.

Missing source file doc comment::
Warns on missing doc comments on source files, this is mainly applied to implicit struct files and namespaces that are re-exported and exposed to the API surface, or reached from the module root. It should properly say that it's missing the doc comment for the namespace or the struct (file) respectively. This should only be thrown when it's clear that the source file is being exposed and not just a single part of it.

=== Blank doc comment

Warns on comments that are present but *fully* blank.

For guidance on how to write good documentation comments, see {zig-docs}#Doc-Comment-Guidance[Zig’s documentation comment guidance].

==== Specializations

To help with clarifications and context, the following cases are reported with a more specific diagnostic message.

Blank module doc comment::
Warns on blank doc comments on the module root source file.

Blank source file doc comment::
Warns on blank doc comments on source files, this is mainly applied to implicit struct files and namespaces that are re-exported and exposed to the API surface. Whole-module re-exports (`pub const ns = @import("enums.zig")`) resolve blank `//!` on the imported file; member re-exports (`pub const Color = @import("enums.zig").Color`) do not.

=== Missing doctest

Allows declarations to have missing doctests.

.What it checks:
* Collects top-level `pub fn name(...)` declarations.
* Collects identifier-style tests `test name { ... }`.
* Emits a diagnostic when a public function has no matching identifier-style doctest.

=== Private doctest

Allows a doctest to reference a private declaration.

.What it checks:
* Collects top-level public function names and public variable/constant names.
* For each `test name { ... }`, emits a diagnostic if `name` is not public.

=== Doctest name mismatch

Warns when a test is likely intended as a doctest but won't be recognized as one.

NOTE: Only checks declarations within the same source file. Cross-file cases are out of scope, mismatched imports are compile errors, and cross-file ambiguitiy makes reliable suggestions infeasible.

.Checks:
String literal match:: `test "foo"` where `pub fn foo(...)` exists – suggests rewriting as `test foo`.
Casing mismatch:: `test Foo` where `Foo` has no matching public declaration but `foo` does — suggests rewriting with the correct casing.

.Checks:
String literal match:: When a public declaration exists with the same name as a string-literal test, suggests rewriting it as an identifier-style test instead.
Casing mismatch:: When an identifier-style test has no matching public declaration but one exists under a different casing, suggests the correctly cased identifier.

.Checks:
String literal match:: A test that matches an existing public declaration by identifier — suggests rewriting it as a doctest.
Casing mismatch:: A doctest identifier that has no matching public declaration
but a case-variant of it does — suggests the correct casing. Currently limited
to capitalization differences; snake_case normalization may be added later.

=== Starts with leading pharse

Warns if a doc comment doesn't start with the required leading phrase.

The leading phrase structure is defined as:

  [<article>] [<identifier kind>] <identifier> [<identifier kind>]...

.Where
Article:: An optional English article (e.g. "a", "an", "the").
Identifier kind:: An optional type of the identifier being documented (e.g. "function", "struct").
Identifier:: The name of the symbol that's being documented.

.Possible leading phrases
====
Module root::
* Module Foo provides...
** The module name check is case insensitive, so "Module foo provides..." is also valid.
* The `foo` module...
** The module name can also be wrapped in backticks.
* Library Foo offers...
** Aside "Module", "Library" can also be used, only when the module is truly a library, meaning it doesn't contain any reserved entry points (e.g. `main`, `build`).

Namespace:: Namespaces can be categorized as a special case of modules, something like sub-modules, so they inherit the same checks and options.
* Namespace JSON offers...
** As well, the module name check is case insensitive.
* The `json` namespace...
** The module name can also be wrapped in backticks.
** Unlike modules, namespaces are only expected to be "namespace". And to help differentiate them from modules or declarations, "Namespace" or "<article> namespace" is always expected.

Declarations::
* Function `foo` does...
* InitOptions represents...
** `InitOptions` being a struct.
* The ParseError error set
** There will be special cases where "error set" could be only "set", if the identifier includes "error" to avoid redundancy.
** When possible to be documented, error values will be expected as: "error tag", "error value", "error member", etc.
* pi represents the mathematical constant...
** `pi` being a global constant or variable.
====

.Options:
* Whether to require always an article or keep it optional (default).
* Whether to always require backticks, optional (default) or never allow them.
* Whether to require the identifier kind to be present or not.
** If required, the position can also be configured, whether to enforce it always before or after, or just allow both (default).
* Identifier presence isn't configurable, it's always expected to be present and match the identifier case.
** Edge cases for false positives and errors should be considered with literal identifiers (e.g. `const @"foo bar" = 1;`).

Warns if doc comments have trailing blank lines.

== Style

=== Identifier case

Checks for consistent identifier naming case conventions, by default it follows the {zig-docs}#Names[Zig documentation style]. It is reported by the `docent style` sub-command rather than the default lint run. Like the complexity checks, it measures *every* identifier — both public and non-public. File discovery starts from the project's module roots on the public API surface, then follows a *reachability traversal* that includes every local `@import("…zig")` (including private imports such as `const Severity = @import("../../severity.zig");`); within each reached file no visibility filter is applied.

Checks are:
`snake_case`::
* Namespaces, considering that every Zig source file is basically a structure, to ease the differentiation between a structure file and a namespace file, without depending on the file name case or its identifier, if a source file doesn't have structure fields, it's automatically marked as a namespace.
* Global variables and constants
* Enumerators
* Fields or values from structures, union, and error sets.
`camelCase`::
* Functions (concrete ones).
`PascalCase`::
* Structures
* Unions
* Enumerations
* Error sets
* Generic functions.

[NOTE]
====
* Error set values technically fall under the value category, similar to enumerators. And Zig itself isn't quite consistent with {zig-docs}std/#std.builtin.OptimizeMode[its own conventions]. Regardless, we'll follow the standard library conventions, so error-set values are expected to be `PascalCase` (e.g. `error.OutOfMemory`).
* It must be careful on re-exported declarations that work as aliases, for example a function that is re-exported in the module root, but lives in a different place, it's a global constant, but internally, it's a function being aliased to that constant. Mostly to avoid false positives, this causes `camelCase` global constants that are in fact just aliases to functions. Declarations whose initializer is a plain identifier, field access, call, or `@import(...)` are therefore skipped.
* Generic functions are detected as those returning `type`.
* Quoted identifiers (`@"foo bar"`) are exempt.
* `@import("…zig")` paths: when the resolved file is a namespace (no structure fields at file scope), the basename must be `snake_case` (e.g. `reachability.zig`, suggested as `diagnostic_message.zig` from `DiagnosticMessage.zig`). The import binding must also be `snake_case` for whole-module imports (e.g. `const severity = @import("severity.zig");`, not `const Severity = …`). Member re-exports (`pub const SeverityLevel = @import("severity.zig").Level;`) name the imported declaration and are not checked as namespace bindings. Struct-at-file-scope modules (e.g. `Diagnostic.zig`, `LintResult.zig`) keep `PascalCase` basenames and bindings.
====

=== Function size

Function size is measured in terms of physical true LoCs, which excludes comments, blank lines, function declaration, and braces.

.Affected by the following factors:
* Comments
* Blank lines
* True LoC, the code per se
* Braces (Zig's defaults to inline opening brace and newline closing brace)
* Function signature, the whole declaration of the function (e.g. `fn hello()`)

Logical LoCs, which counts the number of statements or expressions, isn't considered here, as there are already complexity checks (<<_cyclomatic_complexity>> and <<_cognitive_complexity>>).

== Complexity

=== Cyclomatic Complexity

Defaults to a limit of 10 linearly independent paths, as suggested by <<mccabe-guidance>>. A score of 10–15 is considered _complex_, and anything above 15 is considered _risky_.

NOTE: Unlike tools like Lizard—which can be configured to treat an entire `switch` statement as a single branch—this tool enforces the traditional mathematical definition of _Cyclomatic Complexity_. While some teams modify CC to act as a proxy for readability, it should be used strictly as a **testability metric** (mapping directly to the number of required unit tests). Readability and maintainability are instead handled by **Cognitive Complexity**, which provides a much more reliable metric for human code comprehension.

.Edge cases:
* Exhaustive `switch` statements over `enum`s are still penalized, as each branch represents a real, testable path regardless of whether the compiler requires it.

.Options:
* The threshold can be configured with whole numbers.

=== Cognitive Complexity

Defaults to the Sonar-recommended threshold of 15. See <<sonar-reasoning>> and <<sonar-paper>>. A function is flagged when its score is *strictly greater* than the threshold. It is reported by the `docent complexity` sub-command rather than the default lint run.

Unlike the doc-comment rules, complexity checks measure *every* function — both public and non-public. File discovery still starts from the project's module roots and follows the public API surface, but within each reached file no visibility filter is applied.

Unlike <<_cyclomatic_complexity>>, the score is not a count of paths but an estimate of how hard the control flow is to *understand*. It accumulates three kinds of increments per function:

Structural::
Control-flow structures that also receive a nesting increment: `if`, `for`, `while`, `switch`, and `catch`. Each adds `1 + current_nesting`, so the same structure costs more the deeper it is nested.
Hybrid::
`else` and `else if` add `1` but never receive a nesting increment (the cost of reading the `if` has already been paid), yet they still raise the nesting level for their body.
Fundamental::
Breaks in linear flow that are independent of nesting: each new sequence of binary logical operators (`and` / `or`), a labeled loop `break`/`continue`, and direct recursion.

.Zig-specific mappings:
* `catch` is treated like an exception `catch` clause (structural). `orelse` is the optional counterpart of null-coalescing and is intentionally ignored.
* A labeled `break`/`continue` only increments when the label targets a `for`/`while` loop. Labeled block breaks (`break :blk value`) are value expressions and are not penalized.
* Only direct recursion (a function calling itself by name) is scored; indirect recursion is out of scope.
* `switch` and all of its prongs combined incur a single structural increment, matching the Sonar treatment of `switch`.

.Options
* The threshold can be configured with whole numbers in project config under `complexity.cognitive_threshold`; it defaults to 15.
* The severity (`cognitive_complexity`) follows the usual rule severities and can be set in project config.

=== Maximum function parameter

The maximum number of parameters in many tools and teams often default to around 5, to encourage simpler APIs. In Zig, however, it is common to pass interface parameters such as allocators, writers, and I/O interfaces as explicit dependencies. To be gentler on Zig codebases and more focused on true domain-specific parameters, the default limit here is set to 7.

A function is flagged when its parameter count is *strictly greater* than the threshold. It is reported by the `docent complexity` sub-command rather than the default lint run. Like the other complexity checks, it measures *every* function in the import-closure reachable from the module roots.

.Options
* The threshold can be configured with whole numbers in project config under `complexity.max_fun_params_threshold`; it defaults to 7.
* The severity (`max_fun_params`) follows the usual rule severities and can be set in project config.

== Configuration of rule options

I have my own libraries for it, but JSON is part of the standard library, so that for now. Ideally it should be TOML for simplicity.
Configuration options for the rules are defined in the project root, under the `.config` directory, in the file `docent.json`. It wont be Zig's own notation (ZON) due to lack of JSON schema support.

When no config file is found, or a rule is omitted, Docent uses the `RuleSet` defaults from the library.

.`docent.json`:
====
[source,json]
----
{
  "rules": {
    "missing_doc_comment": "deny",
    "missing_doctest": "allow",
    "private_doctest": "warn"
  },
  "complexity": {
    "cognitive_threshold": 15,
    "max_fun_params_threshold": 7
  },
  "docs": {
    "require_function_param_docs": false
  }
}
----
====

Severity values are `allow`, `warn`, `deny`, and `forbid`. A `forbid` level cannot be relaxed by a later entry in the same file.

The `complexity` block holds the numeric options for the complexity sub-command rules (for example `cognitive_threshold`, `max_fun_params_threshold`). Omitted options fall back to the library defaults.

The `docs` block holds options for documentation rules. Set `require_function_param_docs` to `true` to make `missing_doc_comment` also require `///` on each named function parameter (disabled by default).

.Options:
* You can specify a custom path for the configuration file with `--config-path PATH` on `docent`, `docent status`, `docent complexity`, and `docent style`.
