= Linting Rules

Several rules are inspired by established Go documentation conventions, particularly those enforced by <<godoc-comments>> and <<godoc-lint>>. Go's standard library is notable for its strict, consistent documentation guidelines, and its tooling ecosystem — especially <<golangci-lint>>'s aggregation approach — reflects a culture of broad, opinionated quality checks across style, complexity, and documentation simultaneously.

Rust's Rustdoc and Clippy also informed specific rules here, though their documentation linting tends to focus narrower: missing docs and broken intra-links, etc.. Both ecosystems influenced Docent's rule design.

While default rules focus on documentation comments, style and complexity rules are also available as explicit sub-commands. The reasoning is mostly separation of concerns, as well as keeping consistency with other ecosystem's tooling.

== Doc comments

By default, these rules will be enforced across the public API surface. One can also configure the rules selectively or completely if desired to consider private declarations as well.

=== Missing doc comment

Warns on missing doc comments on public declarations. Including, but not limited to:

* {zig-docs}#Functions[Functions and its parameters]
* {zig-docs}#Variables[Global variables]
* {zig-docs}#enum[Enumerations and its fields]
* {zig-docs}#struct[Structures and its fields]
* {zig-docs}#union[Unions and its fields]
* {zig-docs}#Errors[Error sets]
** Error set fields aren’t checked since Zig documentation generator doesn’t support them.

==== Re-exported declarations

Public re-exports are resolved transitively.

For `pub const Foo = @import("other.zig").Bar`, the rule follows the re-export chain and checks the documentation on the final resolved declaration.

* If the resolved declaration is documented, no diagnostic is emitted.
* If it is undocumented, the diagnostic points to the resolved declaration.
* If resolution fails, the re-export is skipped to avoid false positives.

==== Specializations

To help with clarifications and context, the following cases are reported with a more specific diagnostic message.

Missing module doc comment::
Warns on missing doc comments on the module root source file.

Missing source file doc comment::
Warns on missing doc comments on source files, this is mainly applied to implicit struct files and namespaces that are re-exported and exposed to the API surface, or reached from the module root.

=== Blank doc comment

Warns on comments that are present but *fully* blank.

For guidance on how to write good documentation comments, see {zig-docs}#Doc-Comment-Guidance[Zig’s documentation comment guidance].

=== Missing doctest
Allows declarations to have missing doctests.

.What it checks:
* Collects top-level `pub fn name(...)` declarations.
* Collects identifier-style tests `test name { ... }`.
* Emits a diagnostic when a public function has no matching identifier-style doctest.

=== Private doctest
Allows a doctest to reference a private declaration.

.What it checks:
* Collects top-level public function names and public variable/constant names.
* For each `test name { ... }`, emits a diagnostic if `name` is not public.

=== Doctest name mismatch
Warns when a test is likely intended as a doctest but won't be recognized as one.

NOTE: Only checks declarations within the same source file. Cross-file cases are out of scope, mismatched imports are compile errors, and cross-file ambiguitiy makes reliable suggestions infeasible.

.Checks:
String literal match:: `test "foo"` where `pub fn foo(...)` exists – suggests rewriting as `test foo`.
Casing mismatch:: `test Foo` where `Foo` has no matching public declaration but `foo` does — suggests rewriting with the correct casing.

.Checks:
String literal match:: When a public declaration exists with the same name as a string-literal test, suggests rewriting it as an identifier-style test instead.
Casing mismatch:: When an identifier-style test has no matching public declaration but one exists under a different casing, suggests the correctly cased identifier.

.Checks:
String literal match:: A test that matches an existing public declaration by identifier — suggests rewriting it as a doctest.
Casing mismatch:: A doctest identifier that has no matching public declaration
but a case-variant of it does — suggests the correct casing. Currently limited
to capitalization differences; snake_case normalization may be added later.

=== Starts with leading pharse

Warns if a doc comment doesn't start with the required leading phrase.

The leading phrase structure is defined as:

  [<article>] [<identifier kind>] <identifier> [<identifier kind>]...

.Where
Article:: An optional English article (e.g. "a", "an", "the").
Identifier kind:: An optional type of the identifier being documented (e.g. "function", "struct").
Identifier:: The name of the symbol that's being documented.

.Possible leading phrases
====
Module root::
* Module Foo provides...
** The module name check is case insensitive, so "Module foo provides..." is also valid.
* The `foo` module...
** The module name can also be wrapped in backticks.
* Library Foo offers...
** Aside "Module", "Library" can also be used, only when the module is truly a library, meaning it doesn't contain any reserved entry points (e.g. `main`, `build`).

Namespace:: Namespaces can be categorized as a special case of modules, something like sub-modules, so they inherit the same checks and options.
* Namespace JSON offers...
** As well, the module name check is case insensitive.
* The `json` namespace...
** The module name can also be wrapped in backticks.
** Unlike modules, namespaces are only expected to be "namespace". And to help differentiate them from modules or declarations, "Namespace" or "<article> namespace" is always expected.

Declarations::
* Function `foo` does...
* InitOptions represents...
** `InitOptions` being a struct.
* The ParseError error set
** There will be special cases where "error set" could be only "set", if the identifier includes "error" to avoid redundancy.
** When possible to be documented, error values will be expected as: "error tag", "error value", "error member", etc.
* pi represents the mathematical constant...
** `pi` being a global constant or variable.
====

=== Missing summary terminal punctuation
Warns if, within a doc comment, the first and foremost paragraph (summary) doesn't end with a punctuation mark.

.Checks:
Last character:: Checks if the last character is a punctuation mark (e.g., period, exclamation mark, question mark).

=== Trailing empty lines in doc comments
Warns if doc comments have trailing empty lines.

== Style
=== Naming conventions
Naming conventions for declarations and module (file names) follow the {zig-docs}#Names[Zig documentation].

.With the following clarifications:
* Error set values are expected to be snake_cased, even though the standard library uses PascalCase for them, they technically fall under the value category, similar to enumeration values. ==== Tiger Beetle's Tiger Style
Tiger Beetle has its own style convention called https://tigerstyle.dev/[Tiger Style], and it's a nice reference point to consider for _experience_:

* https://tigerstyle.dev/#nouns-and-verbs[Noun and Verbs]
* https://github.com/tigerbeetle/tigerbeetle/blob/main/docs/TIGER_STYLE.md#naming-things[Naming things (Markdown version, more detailed)]

It's also worth considering for its 3 main points: _Safety, Performance, and Experience_.

=== Function size

Function size is measured in terms of physical true LoCs, which excludes comments, blank lines, function declaration, and braces.

.Affected by the following factors:
** Comments
** Blank lines
** True LoC, the code per se
** Braces (Zig's defaults to inline opening brace and newline closing brace)
** Function signature, the whole declaration of the function (e.g. `fn hello()`)

Logical LoCs, which counts the number of statements or expressions, isn't considered here, as there are already complexity checks (<<_cyclomatic_complexity>> and <<_cognitive_complexity>>).

NOTE: This doesn't mean that it will never be implemented, but it's implementation effort will only be considered on demand and contribution.

== Complexity

=== Cyclomatic Complexity
Defaults to a limit of 10 linearly independent paths, as suggested by <<mccabe-guidance>>. A score of 10–15 is considered _complex_, and anything above 15 is considered _risky_.

NOTE: Unlike tools like Lizard—which can be configured to treat an entire `switch` statement as a single branch—this tool enforces the traditional mathematical definition of _Cyclomatic Complexity_. While some teams modify CC to act as a proxy for readability, it should be used strictly as a **testability metric** (mapping directly to the number of required unit tests). Readability and maintainability are instead handled by **Cognitive Complexity**, which provides a much more reliable metric for human code comprehension.

.Edge cases:
* Exhaustive `switch` statements over `enum`s are still penalized, as each branch represents a real, testable path regardless of whether the compiler requires it.

=== Cognitive Complexity
Defaults to the Sonar-recommended threshold of 15. See <<sonar-reasoning>>.

=== Function parameter count

Parameter count limits in many tools and teams often default to around 5, to encourage simpler APIs. In Zig, however, it is common to pass interface parameters such as allocators, Writergate and I/O interfaces as explicit dependencies. To be gentler on Zig codebases and more focused on true domain-specific parameters, the default limit here is set to 7.
